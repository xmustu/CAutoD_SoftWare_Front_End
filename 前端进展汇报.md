## 2. 前端轻量化与 AiMessage 负载优化

结合 `pretext` 库的“分离测量缓存与热路径计算，规避无意义的高频 Reflow”的核心思想，我们排查了当前的大会话渲染逻辑，并落地了如下优化：

**已经落地的优化（Refactor 阶段）：**
- **抽离热路径的类型推断与图片去重**：之前 `AiMessage` 中的 `inferMessageType` (处理大量 includes 检测) 和 `uniqueImages` （Set 对象生成及图片去重）完全在 React 组件的 `useMemo` 或者 render block 中定义，尤其是随着上下文变长、流式逐字 update 时开销会被放大。
- 我们已经将其分离成纯函数架构至 `src/utils/messageUtils.js` (`inferMessageType`, `extractUniqueImages`, `safeParseMetadata`)，使每次 React render 中的引用与对象创建趋向于最简。

**针对 `pretext` 轻量化适配的可行建议**：
1. **纯粹测算预测与分层渲染**：当后续长文本（Markdown）、富代码块导致严重抖动时，可在完成流式输入前使用轻量化 Text 元素兜底，在结束流后再真正走 Markdown parsing/React-Syntax-Highlighter (类似 `pretext` “先布局，后正式装载”的思路)。
2. **纯算术虚拟预估**：`AiMessage` 可在服务器明确返回行高/行数 meta 信息的场景下，跳过 DOM 测量机制预分配高度，减少 React 的 Layout Thrashing。

## 5. 测试覆盖说明 (本次补充)
- **测试框架**：Vitest + React Testing Library (RTL) + jsdom
- **当前测试文件数**：1 个 (`ConversationDisplay.test.jsx`)，涵盖核心的会话消息分发与渲染逻辑。
- **当前通过用例数**：5 个用例全部绿灯通过 (`3 passed -> 5 passed`)。
- **测了什么**：
  1. 不带过滤时的默认消息合并展示。
  2. `filterTaskType="optimize"` 分支下对优化日志消息的精准保留。
  3. `filterTaskType="geometry"` 分支下对三维建模反馈的精准保留。
  4. **[回归新增] 保护携带 Markdown Table 的普通对话**：验证当对话内容含有易混淆的关键词且附带 Markdown 表格时，推断算法不会将其误判/退化进入 `optimize` 剥离渲染器的纯文本视图。
  5. **[回归新增] Geometry 降级提示 UI**：验证当系统仅输出了 `cad_file`（如 `.step`）而缺少可借助浏览器直接渲染的 `stl_file` 时，界面能正确反馈 `“当前前端仅支持 STL 预览，CAD 文件可下载但不可直接预览”` 的保护性警告，避免卡在永久 Loading 态。
- **怎么测的**：
  通过 `vitest` + `jsdom` 在 Node 环境隔离构造 `mockMessages` 数组，挂载 `<ConversationDisplay>` 组件后使用 `@testing-library/react` (RTL) 断言指定文本和条件降级 UI 容器在 DOM 中的出场与隐藏状态。
- **结果是什么**：
  通过对长文本特征判断的放宽以及 UI 异步请求的容错兜底，修复了原有的渲染结构丢失与无限锁死问题，测试用例全程执行耗时约 8.1s，5项指标全部达成 Green 状态隔离。
